---
title: GitHub OIDC Authentication
description: GitHub Actions OIDC authentication - no secrets in workflows
---

The Microsoft 365 provider supports authentication using GitHub Actions' OpenID Connect (OIDC) tokens. This approach allows Terraform to authenticate to Microsoft 365 services directly from GitHub Actions workflows without storing long-lived credentials as GitHub secrets.

## How GitHub OIDC Works

<Steps>
  <Step title="Workflow Permissions">
    The workflow runs with `permissions: id-token: write`, causing the runner to prepare an OIDC token
  </Step>
  
  <Step title="Environment Variables">
    GitHub injects `ACTIONS_ID_TOKEN_REQUEST_URL` and `ACTIONS_ID_TOKEN_REQUEST_TOKEN` into the job environment
  </Step>
  
  <Step title="Provider Configuration">
    The provider automatically reads these environment variables during configuration
  </Step>
  
  <Step title="Request JWT">
    The provider requests a short-lived JWT from GitHub's OIDC provider
  </Step>
  
  <Step title="Exchange Token">
    The JWT is exchanged for an Azure access token via workload federation
  </Step>
  
  <Step title="API Authentication">
    The resulting token is used to authenticate Microsoft Graph API calls
  </Step>
</Steps>

## Prerequisites

- A GitHub repository where you'll run Terraform
- Permissions to create and configure app registrations in Microsoft Entra ID
- Ability to modify GitHub Actions workflows
- Azure CLI installed (for setup commands)

## Setup

<Steps>
  <Step title="Create App Registration">
    ```bash
    # Set variables
    export TENANT_ID="00000000-0000-0000-0000-000000000000"
    export APP_NAME="terraform-provider-microsoft365"
    export GITHUB_ORG="your-github-org"
    export GITHUB_REPO="your-github-repo"
    
    # Create app registration
    APP_ID=$(az ad app create --display-name "$APP_NAME" --query appId -o tsv)
    APP_OBJECT_ID=$(az ad app show --id "$APP_ID" --query id -o tsv)
    
    # Create service principal
    az ad sp create --id "$APP_ID"
    
    # Grant Microsoft Graph permissions
    az ad app permission add \
      --id "$APP_ID" \
      --api 00000003-0000-0000-c000-000000000000 \
      --api-permissions 78145de6-330d-4800-a6ce-494ff2d33d07=Role
    
    # Grant admin consent
    az ad app permission admin-consent --id "$APP_ID"
    ```
  </Step>
  
  <Step title="Configure Federated Credential">
    ```bash
    # For a specific branch
    az ad app federated-credential create \
      --id $APP_ID \
      --parameters "{\"name\":\"github-federated-credential\",\"issuer\":\"https://token.actions.githubusercontent.com\",\"subject\":\"repo:${GITHUB_ORG}/${GITHUB_REPO}:ref:refs/heads/main\",\"audiences\":[\"api://AzureADTokenExchange\"]}"
    ```
  </Step>
</Steps>

## Subject Patterns

| Scenario | Subject Format | Example |
|----------|----------------|---------||
| Specific branch | `repo:ORG/REPO:ref:refs/heads/BRANCH` | `repo:octo-org/octo-repo:ref:refs/heads/main` |
| Any branch | `repo:ORG/REPO:*` | `repo:octo-org/octo-repo:*` |
| Pull requests | `repo:ORG/REPO:pull_request` | `repo:octo-org/octo-repo:pull_request` |
| Specific tag | `repo:ORG/REPO:ref:refs/tags/TAG` | `repo:octo-org/octo-repo:ref:refs/tags/v1.0.0` |
| Specific environment | `repo:ORG/REPO:environment:ENV` | `repo:octo-org/octo-repo:environment:production` |

## GitHub Actions Workflow

```yaml
name: Terraform Microsoft 365

on:
  workflow_dispatch:

# Permission to request the OIDC JWT ID token
permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: "1.13.3"

      - name: Terraform Init
        run: terraform init
        working-directory: path/to/terraform

      - name: Terraform Plan
        run: terraform plan
        working-directory: path/to/terraform
        env:
          M365_TENANT_ID: ${{ secrets.TENANT_ID }}
          M365_AUTH_METHOD: "oidc_github"
          M365_CLIENT_ID: ${{ secrets.M365_CLIENT_ID }}
```

## Provider Configuration

<Tabs>
  <Tab title="Environment Variables (Recommended)">
    ```bash
    export M365_TENANT_ID="00000000-0000-0000-0000-000000000000"
    export M365_AUTH_METHOD="oidc_github"
    export M365_CLIENT_ID="00000000-0000-0000-0000-000000000000"
    ```
    
    ```terraform
    provider "microsoft365" {
      auth_method = "oidc_github"
    }
    ```
    
    <Info>
    `oidc_request_url` and `oidc_request_token` are automatically populated from `ACTIONS_ID_TOKEN_REQUEST_URL` and `ACTIONS_ID_TOKEN_REQUEST_TOKEN`.
    </Info>
  </Tab>
  
  <Tab title="Terraform Configuration">
    ```terraform
    provider "microsoft365" {
      auth_method = "oidc_github"
      tenant_id   = "00000000-0000-0000-0000-000000000000"
      entra_id_options = {
        client_id = "00000000-0000-0000-0000-000000000000"
      }
    }
    ```
  </Tab>
</Tabs>

## Security Best Practices

<Warning>
**Security Recommendations**

1. **Narrow the scope of trust**: Use specific branch, environment, or tag conditions instead of wildcard subjects
2. **Separate app registrations**: Create separate apps for different repositories or workflows
3. **Conditional access policies**: Configure additional conditions in Microsoft Entra ID
4. **Limit API permissions**: Grant only the minimum required permissions
5. **Enable auditing**: Monitor token issuance and usage in Microsoft Entra ID logs
</Warning>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Missing Environment Variables">
    Error: `GitHub OIDC authentication requires oidc_request_url and oidc_request_token`
    
    Ensure your workflow has the `permissions: id-token: write` directive properly configured.
  </Accordion>
  
  <Accordion title="OIDC Provider Not Available">
    Error: `OIDC provider not available`
    
    This happens when trying to use GitHub OIDC authentication outside a GitHub Actions workflow. The method only works within GitHub Actions.
  </Accordion>
  
  <Accordion title="Federated Credential Configuration">
    Error: `No matching federated identity record found`
    
    The subject claim in the JWT doesn't match any federated credential. Double-check that your subject pattern exactly matches your workflow's context.
    
    ```bash
    # List federated credentials
    az ad app federated-credential list --id $APP_ID -o table
    ```
  </Accordion>
  
  <Accordion title="Insufficient Permissions">
    Error: `Authorization_RequestDenied`
    
    The app registration doesn't have the necessary Microsoft Graph permissions. Grant permissions and admin consent:
    
    ```bash
    az ad app permission add --id $APP_ID --api 00000003-0000-0000-c000-000000000000 --api-permissions PERMISSION_ID=Role
    az ad app permission admin-consent --id $APP_ID
    ```
  </Accordion>
</AccordionGroup>
